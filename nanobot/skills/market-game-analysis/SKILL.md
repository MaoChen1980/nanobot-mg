# MGA — Market Game Analysis（新框架 v4.1）

> **核心出发点**：市场价格是由**定价决定性力量**推动的。我们要识别这股力量是谁、想干嘛、什么时候放手，然后趁势而行，不接最后一棒。

## 核心原则

1. **先看数据，后找原因** — 不是先有假设，是先观察到信号，再去调查
2. **因果优先，量价印证** — 因果关系是判断工具，量价关系是印证工具，优先级不能颠倒
3. **识别不了就不做** — 定价决定性力量是入场门槛，不是分析结果
4. **事出反常必有妖** — 量价和因果不匹配时，撤退不需要等确认

## Step 0 — 市场扫描（每天必做）

### 两种模式

| 模式 | 触发条件 | 扫描范围 |
|------|---------|---------|
| **日常监控** | 无特定触发 | 高流动性连续主力关注列表 |
| **全市场确认** | 需要跨品种成交额、增仓或席位排名 | 交易所或已授权的全市场数据源 |

**全市场扫描触发条件（OR，任一符合即扫描）：**

| 指标 | 标准 |
|------|------|
| 涨跌幅 | > 1%（任一方向，盘中候选） |
| 成交额排名 | 前20名（需要同合约乘数和完整市场数据） |
| 增仓量排名 | 前20名（仅盘后官方持仓差） |
| V/OI 排名 | 前10名（仅盘中热度代理，盘后复核） |

> ⚠️ **做空机会不能漏**：触发条件默认找"异常上涨"，容易漏掉做空机会。每次扫描必须同时检查：
> - **跌幅前排**：跌 > 1%，再核对官方成交额与持仓变化
> - **V/OI < 0.5 缩量下跌**：仅提示可能是多头平仓，不能直接判断资金方向
> - **V/OI > 1 放量下跌**：仅提示需要核对空头席位与官方增仓，不是有效做空的充分条件
> - 如果没有成交额大 + 放量下跌的品种 → 结论里明确写"当前做空机会不成熟"，不得默认全做多

**过滤建议**：盘中至少出现“涨跌异常 + 可解释事件”才进入核验；完整市场数据可得时，再用成交额、增仓和席位做二次过滤。

**扫描结果：**

| 结果 | 后续 |
|------|------|
| 有任一指标触发 | 进入 Step 0.1 步骤完整性自检 |
| 无任何触发 | 今天结束 |

## Step 0.5 — 持仓方向校验 ⚠️（每次分析前必做）⚠️

> **⚠️ 强制执行：未执行 Step 0.5 禁止发送行情报告或给出操作建议。**
> **触发条件：** 用户明确提及持仓、仓位、或询问某个持仓品种时。**即使用户未提及，每次分析前也必须主动确认持仓方向。**

### 持仓校验清单

在给出任何操作建议前，必须完成以下校验：

| 校验项 | 操作 |
|--------|------|
| **确认持仓方向** | 用户是多头、空头、还是空仓？方向必须精确到品种 |
| **确认持仓成本** | 用户的开仓价位是多少？当前价位是盈利还是亏损？ |
| **方向一致性检查** | 你的建议方向是否与用户持仓方向一致？ |
| **反向风险提示** | 若建议方向与持仓方向相反，必须明确写出风险 |

### 方向矛盾处理规则

| 场景 | 正确做法 |
|------|---------|
| 用户持有多头，你建议做空 | ❌ 禁止。先说"你的多头持仓建议持有"，再解释为何短线不追多 |
| 用户持有空头，你建议做多 | ❌ 禁止。先说"你的空头持仓已亏损，建议观察/持有"，不追高 |
| 用户空仓，你建议开仓 | ✅ 可以。但必须说清楚是开新仓，不是补现有仓位 |
| 用户持仓方向与你的建议方向一致 | ✅ 可以。给出持有/加仓/止盈建议 |

### 持仓品种逐一覆盖

- **禁止只分析涨跌最大的品种**，忽略用户其他持仓
- 用户持有哪些品种，分析就必须覆盖哪些品种
- 棕榈油在 09:00 有建议但 11:00 未更新 → 必须更新，不能遗漏
- 每个持仓品种都要给出「当前状态 + 建议」

## Step 2 — 什么资金在作妖？

> ⚠️ **盘中版本限制**：新浪连续主力报价**不包含席位数据**。盘中分析时：
> - 不得声称「主力席位加仓」或「空头席位减仓」（无数据支撑）
> - 只能基于 V/OI、成交量、价格走势做代理判断
> - 完整席位分析须在**交易时段后**（15:00 后）获取交易所每日行情数据再执行
> - 报告中必须明确写「盘中数据，席位待盘后确认」

**核心三问（盘中代理判断）：**

```
1. 什么资金在作妖？
   → 成交量特征：放量/缩量？V/OI 高还是低？（盘后替换为席位数据）
   → 价格与持仓联动：增仓上行/减仓下行 → 顺势；增仓震荡/减仓反弹 → 纠结
   → 是产业套保？投机资金？大资金还是散户？

2. 这股资金想干嘛？
   → 建仓？拉升？出货？对敲？
   → 目标价位大概在哪？

3. 这股资金能不能打赢？
   → 这股资金自己的体量够不够？
   → 对手盘是谁？实力如何？
   → 能持续多久？
```

**结果判断：**

| 能识别 | 后续 |
|--------|------|
| 能识别这股资金的类型、意图、体量 | 进入 Step 3 |
| 识别不了 | 结束，不做 |

## Step 4 — 量价印证 ⚠️（action 标签必须来源此步骤）

**因果逻辑和量价关系是否匹配？**

**⚠️ action 标签来源规范：**
- 「可持」「可开多」「加仓」→ Step 4 匹配 + Step 2 资金可识别 + Step 3 知识可匹配
- 「不追空」「不追多」「观望」→ Step 4 匹配但 Step 2/3 有不确定性
- 「撤退」「止盈/止损」→ Step 4 不匹配 或 Step 5 退出信号触发
- 「无法判断」→ 缺 Step 2 席位数据 或 Step 3 知识库无匹配

**禁止：**
- ❌ 用脚本内置逻辑生成 action 标签而不经验证
- ❌ 缺少 Step 2 资金识别就写「主力拉升」
- ❌ 缺少 Step 3 知识匹配就写「历史规律支持」

### 匹配的典型（大资金意图与量价表现对照）

| 因果逻辑 | 匹配的量价表现 |
|---------|--------------|
| 大资金想拉升 | 放量 + 增仓 + 价格上涨 |
| 大资金想出货 | 放量 + 增仓 + 价格不涨或微涨（派发） |
| 大资金想建仓 | 缩量 + 价格区间震荡（吸筹） |
| 空头被迫平仓 | 持仓量降 + 价格涨（空头止损推动） |

### 不匹配的典型

- 成交量翻3倍，价格原地不动
- 新闻利好，但价格暴跌
- 持仓量暴增，价格却横盘
- 缩量上涨，感觉要跌但一直不跌

**结果判断：**

| 匹配状态 | 后续 |
|---------|------|
| 匹配 | 可持仓 / 可开仓 |
| 不匹配 | 触发反向分析 |

### 反向分析（内嵌在 Step 4）

不匹配时触发：
- 这个量价异常背后，有没有没被识别的因果力量？
- **能找到** → 更新因果判断，回到 Step 2 重新验证
- **找不到** → 立即撤退

## Step 5 — 退出信号

**因果关系和量价关系开始背离时撤退**

- 不是等止损位到了才跑
- 不是等价格开始跌了才跑
- 是**因果和量价开始对不上**的时候就开始跑

**撤退后：**
- 记录本次行情（时间/品种/因果和量价背离的具体表现）
- 用真实结果验证当初的因果判断
- 更新品种知识库

## 数据获取

### 盘中候选（连续主力报价）

运行 `node nanobot/skills/market-game-analysis/references/market-scan.mjs`（cwd=`/Users/chenmao/projects/nanobot-mg`），或以 `node nanobot/skills/market-game-analysis/references/market-scan.mjs SC0 AU0 AG0` 限定品种；`--json` 输出可保存的结构化结果。

- 该脚本只依赖 Node.js 内置 `fetch`，请求新浪连续主力报价并用报价中的昨结字段计算涨跌。
- 报价中的成交、持仓和 V/OI 仅为热度代理；不得据此计算增仓、成交额排名或会员资金流。
- **数据时效校验（强制）：** 脚本输出 JSON 含 `hasStaleData`、`staleCount`、`staleThresholdMinutes` 字段。当 `hasStaleData === true` 时，报告中必须明确标注「⚠️ 数据定格 >15 分钟，置信度降级」，不得基于陈旧数据给出确定性操作建议。
- CFFEX 品种不使用此通道；广期所无夜盘品种须检查报价时间，15:00 定格数据不算夜盘信号。

### 盘后确认与完整信息

按 `references/data-sources.md` 的顺序补齐：

1. 下载交易所每日行情，用同一合约的官方结算价和持仓量确认涨跌及增仓。
2. 查询前二十会员成交/持仓排名，判断多空集中度和方向；没有公开数据时明确写“无法确认席位”。
3. 查询仓单、库存、供需和宏观发布，解释价格异动；新闻只作解释，不能单独作为资金证据。
数据超时、合约映射不清、或第三方价格与官方结算不一致时，结论只能是"观望/待核验"。

3. 若仍超时，标记 DCE 数据「不可用」，结论中注明「DCE 席位数据缺失，无法确认空头席位」

**禁止：**
- ❌ DCE 数据获取失败时直接跳过该交易所
- ❌ 不尝试 seat-data-tempfile skill 直接用其他交易所数据推断
- ❌ 声称「DCE 数据正常」但未验证临时文件内容

### 原油合约标注规范

**原油数据源区分（防止用户混淆）：**

| 原油品种 | 数据来源 | 合约标注 |
|---------|---------|---------|
| 内盘 SC 原油 | akshare futures_fees_info / 新浪主力报价 | 标注「SC 原油主力」或合约代码（如 SC2409） |
| 外盘 WTI 原油 | qhweb.eastmoney.com/quote/yepan（NYMEX） | 标注「NYMEX WTI 原油」 |
| 外盘布伦特原油 | qhweb.eastmoney.com/quote/yepan | 标注「布伦特原油」 |

**铁律：**
- 报告中必须明确标注原油合约类型（内盘 SC vs 外盘 WTI/布伦特）
- 内盘 SC 原油与外盘 WTI/布伦特存在汇率和品质差异，**不可直接对比价格**
- 用户投诉「原油价格不对」时，首先检查是否将内盘 SC 与外盘数据混淆，必要时主动说明合约类型

**用户投诉响应：**
当用户质疑原油数据时：
1. 检查当前使用的原油数据合约类型
2. 若为内盘 SC 原油，主动说明「当前数据为内盘 SC 原油主力，不同于外盘布伦特/WTI，两者价格不可直接对比」
3. 若确实需要对比，提供外盘数据源的价格作为参考（并标注来源）
