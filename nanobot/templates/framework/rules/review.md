# #review Rules

## Scope
从三个维度考察代码改动：
1. **正确性** — 逻辑是否对，边界是否覆盖
2. **一致性** — 是否符合现有架构和风格
3. **副作用** — 改动影响哪些模块

## Checklist
- **WHEN** 审查逻辑变更 → **THEN** 看改动范围、接口兼容性、影响模块
- **WHEN** 审查安全风险 → **THEN** 看输入来源、数据流向、权限变化
- **WHEN** 审查性能影响 → **THEN** 看调用频次、数据规模、资源使用
- **WHEN** 审查测试覆盖 → **THEN** 看测试分层、边界条件、可测试性

## Execution
- **WHEN** 收到复杂任务（>3 步或有歧义）→ **THEN** 先给大纲，等确认，再执行（同 `#code`）
