## 三层任务框架

### 步骤 0: 任务分级

不需要问用户。按以下依据自判：

- **轻量**（改几处文案、发一条消息、读一个文件查个天气）：不需要此框架，直接做
- **中等**（修已知 bug、加参数、改单个模块的配置）：Phase 1 → Phase 3（前提查证 + 验证，不推演）
- **重型**（跨模块改动、架构决策、排未知故障、涉及数据安全或不可逆操作）：完整三阶段

**判断标准：改错了用户会不会注意到？**
- 不会 → 轻量
- 会，但能快速回滚 → 中等
- 会且难以回滚 → 重型

---

### Phase 1: 前提清点与验证

**核心原则：不要凭记忆假设任何事。凡是你能想到的前提，就是你需要验证的条目。**

验证方法不靠想，靠工具：

1. **任务自身信息**
   - 涉及的文件：必须 `read_file` 读到实际内容（不是凭印象），涉及的所有文件都读
   - 涉及的函数/类/API：读完文件后确认签名和语义符合预期
   - 配置文件、依赖声明、schema 定义：直接读原始文件

2. **改动辐射范围**
   - 所有引用点：grep 调用方，确认每处调用是否会受影响
   - 上下游依赖：被改动的模块被谁导入、导出什么、契约是什么
   - 测试覆盖：相关测试在哪儿，测了什么没测什么

3. **隐式假设**（这是 LLM 最常漏的）
   - "XX 一定存在" → 这是假设，先用 glob/grep 确认存在
   - "XX 是字符串类型" → 读类型定义或运行时日志
   - "这个字段一定有值" → 查数据样例或 schema

4. **环境和约束**
   - 运行时版本、SDK 版本、依赖版本
   - 权限、网络可达性、磁盘空间
   - 构建/测试命令能否在当前环境执行

**标注规则：**
- `[已确认]` — 有工具结果支撑（读过文件、看过日志、跑过命令）
- `[待验证]` — 想到但还没查，立即去查
- `[未知]` — 工具也无法确认的（极少见）

**完成标准：** 所有和任务走向相关的前提 >= [已确认]。有 [待验证] 的关键前提 → 优先查证，不需要问用户。查到信息冲突 → 标注冲突，以此作为推演的基础之一。

---

### Phase 2: 方案推演与减除

基于 Phase 1 的前提集：

- 对每个方案标注依赖哪些前提（引用 Phase 1 条目）
- 标注致命反例：哪个前提证伪会导致方案作废
- **减除无效**：关键前提证伪 → 丢弃该方案，讨论下一个

只有一个明显做法时，说明"依赖的所有前提已确认，无可选替代方案"即可，不强行编方案。

---

### Phase 3: 验证闭环

- **执行验证**：实际跑检查，不是口头说
  - 写改动了 → `read_file` 读回来确认
  - 改了函数签名 → grep 所有调用方确认对齐
  - 改了配置 → 用预期格式解析验证
  - 改完代码 → 跑构建 + 跑编译 + 跑相关测试
- **结果比对**：预期 vs 实际，不一致必须标注
- **失败回退**：验证不通过 → 失败本身就是新的 [已确认] 前提 → 回到 Phase 1 更新前提集 → 重新推演。不原地重试，不绕过，不降标准。
- **成功标准**：所有验证项通过，无新的矛盾信息出现
