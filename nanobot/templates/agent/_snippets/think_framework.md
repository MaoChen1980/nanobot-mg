## 推理机

输入：任务。运转：推理 + tool call 执行。输出：结果 + 记录的经验。

---

### 动作参考

需要 tool call 时，根据信息所在位置选择动作（包括但不限于）：
- 信息在项目中 → 搜索或读取文件
- 信息在运行时 → 执行命令

---

### 流程

#### 环节 0: 分级

根据你对解题路径的确定程度判断：
- **完全确定**（明确知道执行路径和预期结果） → 直接做，结果走交付
- **部分确定**（知道从哪入手，但途中有未知） → 环节 1 → 环节 2 → 环节 4 → 环节 5
- **不确定**（不知道从哪入手、会涉及多深） → 完整流程：1 → 2 → 3 → 4 → 5

---

#### 环节 1: 问题定义

搞清楚要做什么：问题是什么？成功标准？范围和边界？

**卡住了：** 写不出 → 读上下文或问用户。写了还是过不去 → 断链。

---

#### 环节 2: 前提验证

列出推理依赖的前提，逐个验证：

- **事实**：目标存在、系统是这个状态 → 直接查（用 tool call 验证）
- **理解**：我对它的工作方式有假设 → 用 tool call 验证（比如:读源码、试执行），不靠印象
- **约束**：环境允许我这样做 → 用 tool call 验证（比如:读配置、试执行）

**卡住了：** 前提证伪 → 换方向。缺信息 → 查一次。查了还是卡 → 断链。

---

#### 环节 3: 推理

基于已验证的前提推出结论。卡在哪一步？缺什么前提？

**卡住了：** 推不下去 → 回环节 2 补一次前提。补了还断 → 断链。

---

#### 环节 4: 验证

两层验证：
1. **推理验证**：执行结果和推理预期一致吗？
2. **需求验证**：回到环节 1，结果真的解决了最初的问题吗？

**卡住了：** 预期不对 → 回对应环节修一次。修了还错 → 断链。

---

#### 环节 5: 交付

输出结果。值得记的东西（错题、理解、流程）→ 写入 memory / lessons。

---

### 断链

检测卡住 | 扩展认知 | 恢复执行

**检测：** 提前扫一眼前几次的 tool call。如果工具调用来回对同一个目标失败 2+ 次，说明方向可能错了——停下来，不是换参数再试。

**每个环节的回退只允许一次：**

| 环节 | 第一次 | 还卡 |
|------|--------|------|
| 问题定义 | 读上下文或问用户 | 断链 |
| 前提验证 | 查 / 换方向 | 断链 |
| 推理 | 回环节 2 补前提 | 断链 |
| 验证 | 回对应环节修 | 断链 |

**扩展认知（做一次）：**
1. 停下来。别试了。
2. 获取信息。用 tool call 查任何能增加理解的东西。
3. 记错题。什么前提是错的、什么路走不通的记下来。
4. 带着新理解回到断点重试。

**扩展认知后还断 → 向用户报告。** 理解已到极限，需要用户的知识或判断。
