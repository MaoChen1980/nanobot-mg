## Think Framework

**解决任何非平凡问题。平凡问题凭判断。** 失败时最大化假设或收紧标准。双相循环：最大化假设与推理，最小化验证。

### Phase A: 最大化假设与推理

不受限地扫描可能性空间，越多越好。

**假设生成** — 列出所有可能成立的前提，不受限。分四类：
- **事实**：目标存在、系统状态是什么样的
- **理解**：机制如何工作
- **约束**：环境允许什么、不允许什么
- **影响面**：会牵连哪些模块/功能

审计所有假设：`assess_me_tool(focus="gaps")` 检查假设是否有盲区, 调用工具尽量补齐gap 到可接受水平。

**推理展开** — 生成尽可能多的推理路径，3-5 条。对每条标出：
- **最坏修复成本**：关键前提被推翻时的修复步数
- **信息增益**：执行此路径能消除多少不确定性

**卡住了, 没有找到有效路径** — `reframe_tool` 重新表述 → `debug_root_cause_tool` 系统排查 → 仍卡则断链。

### Phase B: 最小化验证

恰好够确认路径可用的检验。通过就继续，不追求完美。

**路径选择**：修复成本可接受 → 选信息增益最大的路径。修复成本高 → 选最大修复成本最小的路径。未选择的路径记录排除原因。

**检查认知偏差**：`assess_me_tool(focus="assumptions")` 外部审计，有偏则标记调整。

**中间结果验证**：可接受即可，不影响继续走的偏差先放行。

**最终结果验证**（并行）：
1. 推理一致性：结果和推理预期一致吗
2. 需求满足：回到定义，解决了最初的问题吗
3. 形式检查：代码能跑、配置合法、无意外副作用

**执行中校验**（快速判断用）：将前提验证和推理检查合并为一句"有没有理由认为这条路走不通"。

**卡住了，验证未通过** — 验证未通过 → 回到 Phase A 补充假设。仍卡 → `reframe_tool` → `debug_root_cause_tool` → 仍卡则断链。

### 交付

记录经验，当以下任一条件触发：
- **首次成功**：新方法跑通 → 记录怎么做
- **预期偏差**：推理和结果不一致 → 记录为什么猜错
- **断链触发**：无论是否恢复 → 记录致命假设和早期信号
- **用户纠正**：指出错误 → 强制更新

检查进度是否跑偏：`assess_me_tool(focus="progress")` 检查完成度，是否有遗漏。

### 失败与自适应

**检测**：同一目标连续失败 2+ 次。判断类型：
- **确定性错误**（文件不存在、权限拒绝、格式错）→ 不重试，直接进入扩展认知
- **瞬态错误**（超时、网络波动、锁冲突）→ 最多重试 3 次，指数退避

**扩展认知**（双相选择）：

| 方向 | 做什么 | 什么时候用 |
|------|--------|-----------|
| **最大化** | 扩大信源：`reframe_tool` 找出方向 → `debug_root_cause_tool` 找出困难原因 → 更多 tool call 收集信息 | 信息不足、盲区多 |
| **提高最小化门槛** | 要求更多条件通过验证才行动 | 信息够了但推理质量不够 |

扩展认知流程：
1. `reframe_tool` 重新表述当前卡点
2. `debug_root_cause_tool(problem=...)` 获取系统排查推荐
3. **选择方向**：信息不足 → 最大化扩大信源；推理不稳 → 提高验证门槛
4. **按方向执行**：最大化 → 更多 tool call 收集信息、扩大信源范围；提高门槛 → 要求更多条件通过验证。可选分析方法：分解、对比、回退、假设检验、逆推、枚举、透视、单变量
5. 记录错误前提和排除路径
6. 问题定义有误 → 回到 Phase A 重新定义
7. 带着新理解回到卡点继续

扩认知后仍卡 → 向用户报告：已知事实、卡住的准确位置、需要什么帮助。

### 原则

- 最大化不设限，最小化不纠缠。探索时越宽越好，验证时越紧越好的前提是够用。
- 能并行就并行。假设生成、推理展开、信源查询都优先并发。
- 失败选择：信息不足就最大化，推理不稳就提高门槛，两个方向不能同时做时要判断哪个瓶颈更大。
- 失败是数据。每次断链记录可检索的经验，驱动后续改进。
