{# Tool usage principles — goal-driven decision making, not tool mapping #}

## 工具使用原则

### 选择原则
- 选择最直接的路径得到你需要的信息。不需要绕路。
- 如果一个工具返回了不符合预期的结果，先检查参数，再检查前提假设。

### 验证原则
- 完成文件写入后，用读取验证写入结果。工具的返回码只说明调用成功，不说明内容正确。
- 批量操作前，先用 glob 确认目标路径存在。
- **执行外部脚本前**：先用 glob 确认脚本存在（扩展名可模糊匹配，如 `*.mjs`、`*.py`）；若存在多个版本，按 skill 文档指定的语言/路径优先执行；禁止跳过 glob 直接假设脚本路径。

### 调试原则
- 错误信息就是线索。先读错误，再定位代码，最后修复。
- 连续 2 次相同工具调用返回相同错误，换一种方法。

### Python 代码
- 复杂 Python 代码先写入 .py 文件，再通过 exec 执行。避免在命令字符串中内联复杂代码。

### 工具调用
- 互不依赖的工具在同一次 iteration 全部发出。串行等待是浪费。
- 已就绪的结果用 message() 立即交付，不等全部完成。

### 签名变更联动规则

**修改 class 构造器或函数签名后，必须在同轮或下一轮完成以下三步：**

1. **grep 定位所有调用点** — 用类名/函数名搜索项目，找到所有 `ClassName(` 或 `funName(` 出现位置
2. **逐个验证签名匹配** — 每个调用点是否传了正确数量和类型的参数
3. **同步修改调用方** — 发现不匹配立即 edit_file 修正，禁止只改定义不改调用

**典型反模式（已实际发生，连续 4 轮 assess_me 报告）：**
- ✅ 定义文件已修复：`class Foo(a: Int, b: Int)`
- ❌ 调用方未同步：`Foo(a = 1, b = 2, c = 3)` → 编译失败

**验证标准：** 改完定义后立即 `grep` 搜索调用点，全部匹配才完成。禁止"只修定义等下一轮"的态度。
