## 步骤 0：根因诊断

**最高原则：能用修根因解决的问题，就不要创建新 skill。**
**通用性原则：修复必须项目无关。** 不能包含具体项目名、类名、文件路径、领域术语 —— 用 `{{ template_var }}` 占位或抽象为通用概念。项目特化内容写入 `memory/`。

**按顺序检查（1→2→3→4→5），匹配即停。对跳过步骤记录排除原因。** 宁可漏修，不将同一 root cause 归入多个层次。

不要仅凭 {{ problem_source or "描述" }} 做判断。必须用工具读取实际文件来验证你的诊断。

---

### 诊断流程（按顺序检查，匹配即停）

#### 1. 这是框架/工具 bug 吗？

**本质：** 代码能力不足，不是指令没说清。修复后所有用户受益。

**判断标准：** 行为问题能否通过修框架代码（`loop.py`、`runner.py`、工具等）来解决？

典型特征：工具返回错误格式、行为与预期不一致、缺少必要参数、组件间格式不兼容。

**关键自问（排除时验证）：**
> 「如果 system prompt 写得再完美，这个问题能解决吗？」
> - **不能 → L1，修代码**
> - **能（加条规则就行）→ 代码能力够，往下走 L2**

**确认方法：** 用 `read_file`/`grep` 查看相关代码，验证行为是否与预期一致。代码路径：`{{ nanobot_path }}/agent/`（框架）和 `{{ nanobot_path }}/agent/tools/`（工具）。

**→ 如果是：** {{ action_tool_bug or "修复框架代码" }}

---

#### 2. 这是通用行为约束吗？

**本质：** 指令没写对，不是代码能力不够。修正后所有场景受益。

**判断标准：** 行为模式能否抽象为一条适用于所有任务的通用规则或流程？

**优先用通用指令覆盖，避免为每个领域创建 skill。** 一条通用规则覆盖多个场景，不需要每个写一个 skill。

**⚠️ 关键区分：领域特化问题 ≠ 通用行为约束。** 只发生在特定场景（如 MGA cron、期货分析）的是领域规则，走分类 3 修 SKILL.md，不修通用框架。

**典型误判示例：**
- ❌ "MGA cron 必须执行 OUTPUT GATE 校验" → 领域规则，修 `market-game-analysis/SKILL.md`
- ✅ "所有任务都必须先 skill_search 再 read_file" → 通用约束，修框架

**可用模板路径（用 `read_file`/`glob` 读取）：**
- `{{ nanobot_path }}/templates/agent/_instructions/` — 框架指令块
- `{{ nanobot_path }}/templates/agent/_snippets/` — 代码片段
- `{{ nanobot_path }}/templates/agent/system_prompt.md` — 系统 prompt 主模板
- `{{ nanobot_path }}/templates/agent/identity.md` — agent 身份定义
- `{{ workspace_path }}/prompts/*.pt` — **运行时渲染快照**，比读源模板更准确

**关键自问（排除时验证）：**
> 1. 「写进 prompt 后 LLM 会正确执行吗？」 → 会则 L2，不会则回头查 L1
> 2. 「放通用 prompt 会让无关场景加载领域知识吗？」 → 会则本质是 skill，走分类 3/4

**确认方法：**
1. `glob` 列出 `{{ nanobot_path }}/templates/agent/` 下相关文件
2. 判断是否已有规则覆盖，或可新增通用规则替代
3. `read_file` 确认描述与当前内容一致
4. 若已有规则但 LLM 仍异常 → 非指令缺陷，重新分类
5. **含具体场景名/MGA/skill 名的 → 领域特化，走分类 3/4**

**→ 如果是：** {{ action_instruction or "修复 prompt/指令" }}

---

#### 3. 内容放错位置了吗？

**本质：** 内容正确但容器不对。修复方向是"移动内容"，不是"新增能力"。

**判断标准：** 1-2 已排除。问题出在**内容所在文件位置不对**，非内容本身缺失或错误。

| 情况 | 现象 | 修复 |
|------|------|------|
| **领域规则混入通用模板** | 通用模板中包含本应属于某 skill 的领域规则 | 移入对应 SKILL.md |
| **通用规则误入 skill** | SKILL.md 中包含通用行为约束 | 移入通用模板 |

**关键自问（排除时验证）：**
> 「移到正确位置后问题自动消失？」 → 消失则 L3，移动内容；不消失则往下走分类 4

**确认方法：**
1. 定位问题所在文件
2. 判断性质：通用规则 vs 领域特有步骤
3. 判断所在文件类型：模板 vs skill
4. 通用模板含领域内容 → 移入 skill；skill 含通用规则 → 移入模板

**→ 如果是：** {{ action_move_content or "移动内容到正确位置" }}

---

#### 4. 这是 skill 内容问题或需要新 skill 吗？

**本质：** LLM 缺的是**领域实践经验**，编码为 skill 后同类工作都受益。

**判断标准：** 1-3 已排除。问题与已有 SKILL.md 内容相关（步骤错、缺内容、条件不对），或需全新 skill。

| 子情况 | 现象 |
|--------|------|
| **已有 skill 的错误或内容缺失** | 触发条件/步骤/内容错误或缺失，但位置正确 |
| **需要新 skill** | 无匹配 skill，且满足三个门控条件 |

**三个门控条件（全部满足才是 skill 问题）：**
1. **抽象门控：** 流程/方法/决策逻辑，非具体文件或代码位置
2. **粒度门控：** 场景级别（完整用例），非操作级别（单一步骤）
3. **不重复：** skill_search 无相似结果，或已有 skill 不完整

**关键自问（排除时验证）：**
> 「删掉这条规则，下次 LLM 还会犯同样错误？」 → 会则 L4；不会或不确定则走 L5
> 「能否写成『在 X 领域做 Y 时通用的步骤/规则』？」 → 能则 L4；不能则 L5（memory）

**确认方法：**
1. `skill_search` 检索，`k=6`；召回结果 `read_file` 全文对比
2. 确认问题描述与 skill 指引不一致
3. 去掉具体名/版本号后，规则在其他同类项目是否仍成立

**→ 如果是：**
- 已有 skill 的错误或内容缺失 → {{ action_skill_error or "更新该 SKILL.md" }}
- 需要新 skill → {{ action_new_skill or "创建新 skill" }}

---

#### 5. 领域知识 → 记录到 memory

**本质：** 以上 1-4 都不符合——就是**缺一条具体信息**。告诉 LLM 这条信息，它就能正确决策。

**判断标准：**
- 具体项目/框架/工具的特定经验，不可泛化到同类项目
- 操作级别的单一步骤，非场景级流程
- **一告诉就懂** — 只需信息，不需额外流程或方法

**关键自问（确认时验证）：**
> 「能否写成『在 X 领域做 Y 时通用的步骤/规则』（skill）？」
> - **不能 → memory，记录具体事实**
> - **能 → 回退到分类 4（skill）**

**确认方法：**
1. 查看 `{{ workspace_path }}/memory/` 下已有文件
2. 自问：记下后 LLM 能检索并用对吗？
3. **边界检查：** 可泛化为「当做 X 时都应 Y」→ 是 skill，不是 memory

**记录格式：** `- **YYYY-MM-DD** — [触发条件]: 具体方案/结论。必要时补充背景和原因。`

**→ 动作：** {{ action_domain_knowledge or "写入 memory" }}

### 诊断层次 → 修复路径对照表

| 诊断步骤 | 本质 | 修复目标 | 关键约束 |
|---------|------|---------|---------|
| 1. 框架/工具 bug | 代码能力不足，不是指令没说清 | `{{ nanobot_path }}/agent/` 下的 `.py` 文件 | 精确修改，禁止覆盖整个文件 |
| 2. 通用行为约束缺失 | 指令没写对，不是代码能力不够 | `{{ nanobot_path }}/templates/agent/` 下的 `.md` 模板 | 精确修改，禁止覆盖整个模板文件 |
| 3. 内容放错位置 | 内容正确但容器不对 | 从模板 → skill，或从 skill → 模板 | 移动内容，不新增也不删除知识 |
| 4. skill 内容问题/新 skill | 缺乏领域实践经验 | built-in/workspace skill 的 SKILL.md | 已有 skill 做精确修改，新 skill 按规范创建 |
| 5. 领域知识 → memory | 就缺一条具体信息 | `{{ workspace_path }}/memory/` 下的 topic 文件 | 合并到已有 topic，不开新文件 |
