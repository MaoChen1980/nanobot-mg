## 步骤 0：根因诊断

**最高原则：能用修根因解决的问题，就不要创建新 skill。**
**通用性原则：修复必须项目无关。** 无论修工具代码还是补指令规则，内容必须是项目无关的通用行为约束。不能包含具体项目名、类名、文件路径、领域术语——这些都需抽象为通用概念或用 `{{ template_var }}` 占位。项目特化内容应该写入 `memory/`，而非框架模板或 skill。

**按顺序检查（1→2→3→4→5），一旦找到匹配立即停止。对每个跳过的步骤，记录排除原因。** 宁可漏修，不要将同一个 root cause 归入多个层次。

不要仅凭 {{ problem_source or "描述" }} 做判断。必须用工具读取实际文件来验证你的诊断。

---

### 诊断流程（按顺序检查，匹配即停）

#### 1. 这是框架/工具 bug 吗？

**本质：** 代码能力不足，不是指令没说清。修复后所有用户、所有场景受益。

**判断标准：** {{ problem_source or "描述" }}的行为问题，可以通过修改 nanobot 框架代码（loop、runner、工具等）来修复？

典型特征：
- 工具或框架返回错误格式
- 行为与文档/预期不一致
- 缺少某个必要参数或功能
- 组件之间的输出格式不兼容

**关键自问（排除此层时的验证）：**
> 「如果 system prompt 写得再完美，这个问题能解决吗？」
> - **不能解决 → 确实是 L1（代码问题），停下，修代码**
> - **能解决（加一条规则就行）→ 说明代码能力够，往下走 L2**

**确认方法：** 用工具查看相关代码，验证当前代码行为是否与预期一致。代码路径：`{{ nanobot_path }}/agent/`（框架）和 `{{ nanobot_path }}/agent/tools/`（工具）。

**→ 如果是：** {{ action_tool_bug or "修复框架代码" }}

---

#### 2. 这是通用行为约束吗？

**本质：** 指令没写对，不是代码能力不够。修正后所有场景的 LLM 行为都受益。

**判断标准：** {{ problem_source or "描述" }}的行为模式，是否可以抽象为一条通用规则或流程，适用于所有任务？

**优先用通用指令覆盖，避免为每个领域创建 skill。** 一条通用规则（如"先验证再修改"）可以覆盖多个领域场景，不需要每个写一个 skill。

**⚠️ 关键区分：领域特化问题 ≠ 通用行为约束。** 如果问题只发生在特定场景（如 MGA cron 任务、期货分析、代码审查），这是领域特有的业务规则，应该走分类 3 修 SKILL.md，而不是修通用框架。

**常见误判示例：**
- ❌ "MGA cron 任务必须执行 OUTPUT GATE 校验" → 这是 MGA 特有的业务规则，应该修 `market-game-analysis/SKILL.md`
- ❌ "期货分析必须先加载 skill" → 这是 MGA 场景的 skill 加载规则，应该修 `market-game-analysis/SKILL.md`
- ✅ "所有任务都必须先 skill_search 再 read_file" → 这是通用框架行为约束，可以修框架

**你无法直接访问运行时动态构建的完整 system prompt。** 但它的源模板和框架指令块在以下路径，你可以用工具读取：
- `{{ nanobot_path }}/templates/agent/_instructions/` — 框架指令块
- `{{ nanobot_path }}/templates/agent/_snippets/` — 代码片段（通过 Jinja2 include 注入到 system prompt 中）
- `{{ nanobot_path }}/templates/agent/system_prompt.md` — 系统 prompt 主模板
- `{{ nanobot_path }}/templates/agent/identity.md` — agent 身份定义
- `{{ nanobot_path }}/templates/agent/` — 其他顶层模板（如 `assess_me.md`、`evaluator.md`、`extractor_analysis.md`、子 agent prompt 等）
- `{{ workspace_path }}/prompts/` — **运行时渲染快照**（`.pt` 文件），包含每次 LLM 调用时实际发送的完整消息。用 `glob *.pt` 找到最近的文件，`read_file` 查看已渲染的系统 prompt 和指令块，比读源模板更准确

典型特征：
- 可以抽象为「当 X 时必须/禁止 Y」形式的通用规则，不依赖具体领域术语
- 修复涉及增/删/改 `_instructions/`、`_snippets/` 或顶层模板中的内容
- **如问题本身就是领域特化的**（只发生在特定场景）→ 不是通用约束，走 skill 判断

**关键自问（排除此层时的验证）：**
> 1. 「如果我把这条规则写进 prompt，LLM 会正确执行吗？」
>    - **会 → 确实是 L2，停下，修 prompt/指令**
>    - **不会（代码能力不够）→ 回头检查是否其实是 L1**
>
> 2. 「这条规则放到通用 prompt 里，会让无关场景也加载不必要的领域知识吗？」
>    - **会 → 说明它本质是领域 skill，不是通用规则，走分类 3/4**
>    - **不会 → 确实是通用行为约束，可以放在 prompt 中**

**确认方法：**
1. 用 `glob` 列出 `{{ nanobot_path }}/templates/agent/` 下的相关文件（包括 `_instructions/`、`_snippets/` 和顶层模板）
2. 判断是否有已有规则覆盖，或是否可以新增一条通用规则替代
3. 用 `read_file` 读相关文件，确认描述与当前内容是否一致（可能缺少、多余、或描述错误）
4. 如果所有文件已有对应规则但 LLM 仍行为异常，则不是指令缺陷，重新考虑分类
5. **重要**：如果问题描述包含具体场景名称（MGA、期货、市场博弈）、具体 skill 名称（market-game-analysis）、或具体业务术语，这是领域特化问题 → 走分类 3/4

**→ 如果是：** {{ action_instruction or "修复 prompt/指令" }}

---

#### 3. 内容放错位置了吗？

**本质：** 内容在正确的位置就不成问题——只是放错了容器。修复方向是"移动内容"，不是"新增能力"。

**判断标准：** 1-2 已排除。问题出在**内容所在的文件位置不对**，而非内容本身缺失或错误。

有两种典型情况：

| 情况 | 现象 | 修复 |
|------|------|------|
| **领域规则混入通用模板** | 某个通用模板（`_instructions/`、`_snippets/`、顶层模板）中包含本应属于某 skill 的领域特化规则 | 移出通用模板 → 放入对应 skill 的 SKILL.md |
| **通用规则误入 skill** | 某个 SKILL.md 中包含对所有任务都通用的行为约束（如"先验证再修改"） | 移出 skill → 放入通用模板 |

**常见误判示例：**
- ✅ "`_instructions/planning.md` 中写了一句『MGA 分析必须包含多空比』" → 领域规则混入通用模板，应该移入 `market-game-analysis/SKILL.md`
- ✅ "`market-game-analysis/SKILL.md` 中写了『使用工具前必须先 read_file 确认文件存在』" → 这是通用行为约束，应该移入 `_instructions/`
- ❌ "SKILL.md 中某个步骤写错了" → 这不是放错位置，是内容错误，走分类 4

**关键自问（排除此层时的验证）：**
> 「如果我把这段内容移到正确的位置（template → skill 或 skill → template），问题就自动消失了吗？」
> - **会消失 → 确实是位置问题，停下，移动内容**
> - **不会消失（内容本身就是错的/缺失的）→ 不是位置问题，往下走分类 4**

**确认方法：**
1. 定位问题内容所在的文件
2. 判断内容性质：是"所有场景通用的规则"还是"某个领域特有的步骤/条件"？
3. 判断它当前所在的文件类型：通用模板 vs skill
4. 如果通用模板包含领域内容 → 移入 skill；如果 skill 包含通用规则 → 移入通用模板

**→ 如果是：** {{ action_move_content or "移动内容到正确位置" }}

---

#### 4. 这是 skill 内容问题或需要新 skill 吗？

**本质：** LLM 缺的不是通用能力，是**某个领域的实践经验**。把这个领域的"手艺"编码成 skill 后，所有做同类工作的 LLM 都受益。

**判断标准：** 1-3 已排除。问题与某个已有 SKILL.md 的内容相关（步骤错、缺内容、条件不对），或需要一个全新的 skill。

有两种子情况：

| 子情况 | 现象 |
|--------|------|
| **已有 skill 内容错误** | skill 的触发条件错误、步骤错误、缺少必要内容，但内容位置正确（不是放错位置） |
| **需要新 skill** | 没有匹配的已有 skill，且 candidate 满足以下三个门控条件 |

**三个门控条件（必须全部满足才是 skill 问题）：**

1. **抽象门控：** 描述的是流程/方法/决策逻辑，而非具体文件或代码位置
2. **粒度门控：** 场景级别（覆盖完整用例），而非操作级别（单一步骤）
3. **不重复：** skill_search 无相似结果，或已有 skill 内容不完整需要合并

**关键自问（排除此层时的验证）：**
> 「如果我删掉这条规则，下次 LLM 在同样领域还会犯同样错误吗？」
> - **会 → 值得记，确实是 skill 问题**
> - **不会或不确定 → 说明它不够通用，往下走分类 5**
>
> **与 project knowledge（分类 5）的区分：**
> 「这条信息能不能写成『在 X 领域做 Y 时通用的步骤/规则』？」
> - **能 → L4（skill），抽象为领域通用流程**
> - **不能 → L5（memory），它是具体事实，放不进去**

**确认方法：**
1. 用 `skill_search` 检索相关功能，`k=6`
2. 对召回的 skill，用 `read_file` 读 SKILL.md 全文对比
3. 确认问题描述与当前 skill 指引是否确实不一致
4. 验证抽象级别：去掉具体项目名、版本号后，这条规则在其他同类项目中是否仍然成立

**→ 如果是：**
- 已有 skill 内容错误 → {{ action_skill_error or "更新该 SKILL.md" }}
- 需要新 skill → {{ action_new_skill or "创建新 skill" }}

---

#### 5. 领域知识 → 记录到 memory

**本质：** 不是代码问题，不是指令问题，不是 skill 问题——就是**缺一条信息**。告诉 LLM 这条信息，它就能正确决策。

以上 1-4 都不符合，但 candidate 包含有价值的领域特化知识。

**判断标准：**
- 具体项目/框架/工具的特定经验，不可泛化到同类项目
- 操作级别的单一步骤而非场景级流程
- 不够抽象或不够场景级，但未来可能有用的信息
- **一告诉就懂** — LLM 不需要额外的流程或方法，只要知道这条信息就能正确决策

**关键自问（确认此层时验证）：**
> 「这条信息能不能写成『在 X 领域做 Y 时通用的步骤/规则』（即一个 skill）？」
> - **不能 → 确实应该走 memory，记录具体事实**
> - **能 → 说明它应该是一个 skill，回退到分类 4**

**确认方法：**
1. 查看 `{{ workspace_path }}/memory/` 下已有文件
2. 自问：「如果我在 memory 里记下这条信息，下次 LLM 能检索到并用对吗？」
3. **边界检查：** 如果这条信息可以泛化为「当做 X 类工作时都应该 Y」，那它应该是分类 4（skill），不是分类 5

**记录格式建议：**
```
- **YYYY-MM-DD** — [触发条件]: 具体方案/结论。必要时补充背景和原因。
```

**后续使用：** 记录到 memory 后，LLM 可在需要时通过 memory_search 自动检索，或在 context 中注入。

**→ 动作：** {{ action_domain_knowledge or "写入 memory" }}

### 诊断层次 → 修复路径对照表

| 诊断步骤 | 本质 | 修复目标 | 关键约束 |
|---------|------|---------|---------|
| 1. 框架/工具 bug | 代码能力不足，不是指令没说清 | `{{ nanobot_path }}/agent/` 下的 `.py` 文件 | 精确修改，禁止覆盖整个文件 |
| 2. 通用行为约束缺失 | 指令没写对，不是代码能力不够 | `{{ nanobot_path }}/templates/agent/` 下的 `.md` 模板 | 精确修改，禁止覆盖整个模板文件 |
| 3. 内容放错位置 | 内容正确但容器不对 | 从模板 → skill，或从 skill → 模板 | 移动内容，不新增也不删除知识 |
| 4. skill 内容问题/新 skill | 缺乏领域实践经验 | built-in/workspace skill 的 SKILL.md | 已有 skill 做精确修改，新 skill 按规范创建 |
| 5. 领域知识 → memory | 就缺一条具体信息 | `{{ workspace_path }}/memory/` 下的 topic 文件 | 合并到已有 topic，不开新文件 |
