# Explicit Communication Model (显式沟通模型)

## 核心原则

沟通损耗无法消除。方案不是设计"完美协议"，而是**让每一方在合作时拥有最大化的决策信息**。

每个参与方有两个责任：
1. **表达 (Express)** — 显式说出自己的任务、意图、能力、边界
2. **引导 (Elicit)** — 引导对方也补全这四个维度

---

## 四维信息模型

每一次交互的信息包应包含四个维度：

| 维度 | 含义 | 典型问题 |
|------|------|---------|
| **任务 (Task)** | 做什么、交付什么、范围 | 具体要完成什么？交付物是什么？ |
| **意图 (Intent)** | 为什么做、什么算好、优先级 | 为什么要做这个？成功的标准是什么？多紧急？ |
| **能力 (Capability)** | 我能提供什么、我需要什么 | 我有什么信息/资源？我需要什么支持？ |
| **边界 (Boundary)** | 限制、阻碍、升级条件 | 有什么限制？卡在哪里？什么情况需要回传？ |

---

## 三种交互场景

### 1. User ↔ Agent（用户与主 Agent）

**Agent 的责任：**
- 用户给出模糊需求时，主动引导用户补全四维信息
- 显式告知用户自己的能力范围和限制
- 交出方案时，也要交出"为什么选这个方案"

**交互示意：**

```
用户: "帮我优化一下那个接口的响应速度"

Agent:
  任务确认: "你说的优化是哪个接口？/api/search 吗？"
  意图追问: "你期望的优化目标是什么——P99 降到多少？还是先解决超时问题？"
  能力了解: "你那边有接口的压测数据吗？或者能提供最近的调用日志？"
  边界确认: "有什么约束——不能动 DB 结构？不能改协议？"

Agent (最终回应):
  任务: "我会分析 /api/search 接口的慢查询和响应链路"
  意图: "目标是 P99 从 2s 降到 500ms，优先解决超时问题"
  能力: "我能做代码分析、SQL profiling、链路追踪。但我没有生产流量数据"
  边界: "我不能直接访问生产环境。你能提供最近一天的日志吗？"
```

### 2. Orchestrator ↔ Worker（协调者与执行者）

**下发信息包：**

```
## Task
具体做什么，交付物是什么

## Intent
这个任务的背景是什么？服务于什么更大的目标？
成功的标准是什么？（什么算"做完了"？什么算"做得好"？）

## Capability (Orchestrator)
我已经知道什么上下文？
我可以提供什么信息/工具/权限？

## Boundary
有什么约束？（时间、范围、技术限制）
遇到什么情况必须回传？
优先级：这个任务和其他任务的关系是什么？
```

**Worker 的回应：**

```
## Task
我理解的任务是……（确认理解一致）

## Capability (Worker)
我能做 X、Y，不能做 Z
我现在有什么上下文
我需要 A 信息 / B 决策

## Boundary
如果遇到……我会回传确认
预计需要……
```

### 3. Worker → Orchestrator（汇报/求助）

```
## Task
我完成了 X 部分，正在做 Y

## Capability
我发现了 A、B 信息
我遇到了 C 问题，尝试了 D 方案但不行

## Boundary
我需要你决定：方案 E 还是方案 F？
或者：我建议用 E，因为……，你同意吗？
```

## 落地要点

### 对用户交互 (framework.md)

User Requirement Management 补充：

> 当用户给出需求时，主动引导用户思考四个维度：
> - **任务**：澄清具体范围和交付物
> - **意图**：理解背后的目标和优先级
> - **能力**：了解用户能提供什么信息/资源/权限
> - **边界**：确认约束条件和限制
>
> 不要"用户说什么就做什么"。帮用户把需求说清楚再开始执行。

### Worker 身份 (identity.md)

> **显式沟通是你有效协作的基础。**
> 接到任务时：先确认你理解了任务、意图、边界。
> 执行中：如果你需要信息、决策、资源，显式说出来。
> 汇报时：不仅说你做了什么，也说发现了什么、还需要什么、建议什么。

### Subagent 协议

下发时附带意图和边界上下文。
汇报时结构化为：完成情况 + 发现 + 需要 + 建议。

---

## 为什么有效

1. **减少猜测** — 话说清楚，接收方不需要猜
2. **早期纠偏** — 意图不一致在前两步就暴露，而不是做完才发现
3. **更好决策** — 协调者知道 Worker 的能力和状态，才能合理分配任务
4. **自文档化** — 每步交互都含完整上下文，不需要事后还原

---

## 代价

- **信息量增加** — prompt 更长
- **需要训练** — Agent 需要被 prompt 塑造成"自动补全"，而不是"问啥答啥"
- **引导不能过度** — 不能变成冗长的模板填写，要自然融入对话

第一条靠 prompt 设计消化。
第二条靠持续使用和 lesson 积累。
第三条靠设计时注意节奏——引导要逐步展开，不是一次全抛。
