# SOUL.md Prompt 设计原则

SOUL.md 本质是通过 system prompt 注入来约束 LLM 行为。以下结论来自实际使用观察。

## 高杠杆 vs 低杠杆

### 有用的

| 类型 | 例子 | 杠杆 |
|------|------|------|
| **决策原则** | "两难时选更简单的方案" | 高 — LLM 经常面临权衡，原则填补指令空白 |
| **否定式约束** | "不要问我'你想怎么做'" | 高 — 明确禁止能消除常见坏习惯 |
| **工作流模板** | "改动前先 grep 找所有调用点" | 高 — LLM 会机械遵循步骤 |
| **角色设定** | "你是资深工程师" | 中 — 影响语气但易被稀释 |
| **价值观宣言** | "代码可读性最重要" | 中 — 太抽象，只在边界案例起作用 |
| **具体到文件的规定** | "修改 auth.py 需要写测试" | 低 — 易过时，context 一长就被遗忘 |

### 无效/负杠杆

- 重复 CLAUDE.md 已有内容 → 被去重或稀释
- 过于抽象的口号 → 占 token 不改行为
- 惩罚性语气 → LLM 没有"恐惧"机制，不生效
- 长篇幅规范 → 注入位置靠后，被更近指令覆盖

## 形式建议

**否定式比肯定式约束力更强。** 同样语义下：

```
不要猜测 API 参数，去读文档。
```

优于：

```
你应该总是查阅官方文档来确认 API 参数的正确性。
```

否定句语义更明确，LLM 更容易"检测到违规"来抑制行为。

## 通用原则

1. **简洁 > 详细** — 每多一句稀释一分
2. **具体胜过抽象** — "不要问'你想怎么做'" 好于 "要自主工作"
3. **否定式优先** — 禁止比倡导更能约束行为
4. **靠近决策点** — 约束高频决策场景的指令效果最好
